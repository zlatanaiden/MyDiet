# MyDiet 后端完整实施方案

## 技术栈

- **运行时**: Node.js >= 18
- **框架**: Express.js
- **语言**: TypeScript
- **ORM**: Prisma（数据库建模、迁移、查询）
- **数据库**: PostgreSQL
- **认证**: JWT (jsonwebtoken) + bcrypt
- **文件上传**: Multer
- **校验**: Zod（请求参数校验）
- **环境变量**: dotenv
- **日志**: morgan
- **跨域**: cors

## API 设计

所有接口前缀 `/api`，需认证的接口标注 `[Auth]`。

### 1. 认证 (`/api/auth`)

| 方法 | 路径 | 说明 | 请求体 |
|------|------|------|--------|
| POST | `/register` | 注册 | `{ name, email, password }` |
| POST | `/login` | 登录，返回 JWT | `{ email, password }` |

**返回格式**:
```json
{
  "token": "eyJhbG...",
  "user": { "id", "name", "email" }
}
```

### 2. 用户资料 (`/api/users`) [Auth]

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/me` | 获取当前用户完整资料（含 profile、allergies、restrictions、stats） |
| PUT | `/me/profile` | 更新资料（age, gender, height, weight, targetWeight, goal, activityLevel） |
| PUT | `/me/allergies` | 更新过敏源列表 `{ allergies: ["Peanuts", "Shellfish"] }` |
| PUT | `/me/restrictions` | 更新忌口列表 `{ restrictions: ["Low Sodium"] }` |
| PUT | `/me/unit` | 切换单位 `{ unit: "metric" | "imperial" }` |
| PUT | `/me/privacy` | 隐私设置 `{ profilePublic: true | false }` |

**与前端 `UserProfile` 对应关系**:
```
前端 AppContext.user  ←→  GET /api/users/me
前端 setUser()        ←→  PUT /api/users/me/profile
前端 user.allergies   ←→  PUT /api/users/me/allergies
前端 user.restrictions←→  PUT /api/users/me/restrictions
前端 unit             ←→  PUT /api/users/me/unit
```

### 3. 问卷 (`/api/questionnaire`) [Auth]

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/submit` | 提交问卷（首次填写），自动生成周计划 |
| GET | `/status` | 查询是否已完成问卷 |

**请求体**（对应前端 PlanPage 问卷表单）:
```json
{
  "age": 22,
  "gender": "Male",
  "height": 178,
  "weight": 75,
  "targetWeight": 68,
  "goal": "lose",
  "activityLevel": "Moderate",
  "allergies": ["Peanuts", "Shellfish"],
  "restrictions": ["Low Sodium"]
}
```

提交后，后端自动：
1. 保存用户资料
2. 计算 BMI 和每日营养目标（基于 DRI 表）
3. 调用规划算法生成一周饮食计划
4. 将 `planCompleted` 设为 `true`

### 4. 饮食计划 (`/api/plans`) [Auth]

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/weekly` | 获取当前周计划（7天 × 3餐 + 每餐备选） |
| POST | `/weekly/regenerate` | 重新生成整周计划 |
| POST | `/weekly/:day/swap` | 随机换菜 `{ mealType: "breakfast" }` |
| POST | `/weekly/:day/select` | 选择指定备选 `{ mealType: "lunch", alternativeId: "xxx" }` |

**返回格式**（对应前端 `DayPlan[]`）:
```json
[
  {
    "day": "Monday",
    "meals": {
      "breakfast": { "id", "name", "calories", "protein", "carbs", "fats", "image" },
      "lunch": { ... },
      "dinner": { ... }
    },
    "alternatives": {
      "breakfast": [{ "id", "name", "calories", "protein", "carbs", "fats", "image" }],
      "lunch": [...],
      "dinner": [...]
    }
  }
]
```

**与前端对应关系**:
```
前端 weeklyPlan         ←→  GET /api/plans/weekly
前端 swapMeal()         ←→  POST /api/plans/weekly/:day/swap
前端 selectMealAlt()    ←→  POST /api/plans/weekly/:day/select
前端 resetPlan()        ←→  POST /api/plans/weekly/regenerate
```

### 5. 每日记录 (`/api/daily`) [Auth]

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/:date` | 获取指定日期记录（营养进度 + 三餐状态 + 额外餐食） |
| POST | `/:date/check` | 三餐打卡 `{ mealType: "breakfast" }` |
| POST | `/:date/uncheck` | 取消打卡 `{ mealType: "lunch" }` |
| POST | `/streak` | 每日打卡（连续天数 +1） |
| GET | `/streak` | 获取当前连续天数和今日是否已打卡 |
| POST | `/:date/extra` | 添加额外餐食（从 Identifier 确认） |
| DELETE | `/:date/extra/:mealId` | 删除单个额外餐食 |
| DELETE | `/:date/extra` | 删除当日所有额外餐食 |

**返回格式**（对应前端 `DailyRecord`）:
```json
{
  "date": "2026-02-23",
  "nutrition": {
    "calories": { "current": 1847, "target": 2200 },
    "protein": { "current": 92, "target": 130 },
    "carbs": { "current": 210, "target": 280 },
    "fats": { "current": 58, "target": 73 }
  },
  "meals": {
    "breakfast": { "id", "name", "calories", ..., "checked": true },
    "lunch": { ..., "checked": false },
    "dinner": { ..., "checked": false }
  },
  "extraMeals": [
    { "id", "name", "calories", "protein", "carbs", "fats" }
  ]
}
```

**与前端对应关系**:
```
前端 nutrition + meals   ←→  GET /api/daily/:date
前端 toggleMealCheck()   ←→  POST /api/daily/:date/check
前端 streak              ←→  GET /api/daily/streak
前端 incrementStreak()   ←→  POST /api/daily/streak
前端 extraMeals          ←→  GET /api/daily/:date (extraMeals 字段)
前端 addExtraMeals()     ←→  POST /api/daily/:date/extra
前端 removeExtraMeal()   ←→  DELETE /api/daily/:date/extra/:mealId
前端 removeAllExtra()    ←→  DELETE /api/daily/:date/extra
前端 addNutrition()      ←→  自动：打卡/录入额外餐食时后端计算
```

## 核心业务逻辑

### 营养目标计算

根据用户资料和 `Dietary_Reference_Intakes` 表计算每日目标：

```
输入：age, gender, goal, activityLevel
查询：dietary_reference_intakes 表匹配年龄性别段
输出：target_calories, target_protein, target_carbs, target_fats

如果 goal = "lose"  → target_calories × 0.8
如果 goal = "gain"  → target_calories × 1.2
如果 goal = "maintain" → target_calories × 1.0
```

### 饮食规划算法

```
输入：target_calories, target_protein, target_carbs, target_fats, allergies, restrictions
流程：
  1. 从 recipes 表筛选（排除过敏食材）
  2. 按营养匹配度评分
  3. 分配早/午/晚（热量比 3:4:3）
  4. 每餐生成 2-3 个备选方案（营养相近的替代食谱）
  5. 重复 7 天，尽量不重复
输出：weeklyPlan（7天 × 3餐 + 备选）
```

### 打卡与连续天数

```
POST /api/daily/streak:
  1. 检查今日是否已打卡 → 已打卡返回 409
  2. 检查昨日是否打卡 → 未打卡则 streak 重置为 1
  3. 昨日已打卡 → streak += 1
  4. 更新 last_check_date = today
```

## 错误处理

统一错误响应格式：
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email is required",
    "details": [...]
  }
}
```

| HTTP 状态码 | 含义 |
|-------------|------|
| 200 | 成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 |
| 401 | 未认证（无 Token / Token 过期） |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 冲突（如重复打卡、邮箱已注册） |
| 500 | 服务器内部错误 |
